## 1. 목적 선택 라벨

`personas.ts`·09 §3·12 §3.2와 일치하는 enum 3값에 *UI 라벨*을 입힌다. enum 값(`diet`/`digestion`/`other`)은 *불변*. 라벨·마이크로카피만 변경 가능.

### 1.1 `diet` — 체중·식습관 관리

| # | 라벨 (≤20자) | 마이크로카피 (~30자) | 코멘트 |
|---|------|----------|--------|
| **A (권장)** | **체중·식습관 관리** | **더 천천히 먹고 싶어요** | 결과 라벨 + 행동 표현. "다이어트" 단독 회피(01 §1.5 금지). |
| B | 천천히 먹기 연습 | 체중도 같이 신경 써요 | 행동 우선. "체중"이 작아짐. |
| C | 식사 속도 줄이기 | 체중 관리도 같이 봐요 | 가장 중립적이지만 "체중" 신호 약함. |

→ **권장 A**. "체중·식습관 관리"가 박소연 페르소나의 *결과 라벨*과 직결되고, 마이크로카피의 1인칭 표현 "~먹고 싶어요"가 옵션 G 친근 톤에 맞다.

### 1.2 `digestion` — 소화 문제

| # | 라벨 (≤20자) | 마이크로카피 (~30자) | 코멘트 |
|---|------|----------|--------|
| **A (권장)** | **소화·위 건강** | **위염·소화불량으로 고민이에요** | 결과 라벨 두 단어. 마이크로에 김상훈 페르소나 어휘. "개선" 어휘 회피(과한 효과 약속). |
| B | 소화 문제 개선 | 위염·역류로 고민이에요 | "개선"이 약속처럼 들릴 수 있음. |
| C | 위장이 편해지길 | 자주 더부룩하고 답답해요 | 감각 표현. 마이크로가 김상훈에 잘 맞으나 라벨이 길게 느껴짐. |

→ **권장 A**. "소화·위 건강"이 옵션 G 헤드라인의 *위 건강* 어휘와 일치. 마이크로의 "위염·소화불량" 두 단어로 김상훈·박소연 모두 자기 인식. "개선"·"치유"는 의료 광고법 회피.

### 1.3 `other` — 그 외

| # | 라벨 (≤20자) | 마이크로카피 (~30자) | 코멘트 |
|---|------|----------|--------|
| **A (권장)** | **그 외 / 둘 다** | **마음챙김 식사도 궁금해요** | 한지원 페르소나(혼밥·마음챙김) 흡수. "그 외"가 자연스럽고 "둘 다"가 selection paralysis 해소. |
| B | 마음챙김 식사 | 천천히·집중해서 먹고 싶어요 | 한지원에는 강하지만 "둘 다" 케이스를 못 잡음. |
| C | 기타 | 위·체중 둘 다 또는 그 외 | 가장 짧지만 미시청 위험. |

→ **권장 A**. "그 외 / 둘 다" 표기로 *둘 다 해당* 케이스를 명시 흡수. 마이크로카피로 한지원 페르소나의 마음챙김 동기를 흡수.

## 2. 폼 위 헤드 + helper

| # | 헤드 | 코멘트 |
|---|------|--------|
| A | 어떤 이유로 관심 있으세요? | 가장 부드럽고 강요 없음 |
| **B (권장)** | **어느 쪽이 가까우세요?** | 가장 짧고 간결. 3선택지 *중 가까운 것* 뉘앙스 — 정확한 진단을 강요하지 않음. |
| C | 주된 목적이 무엇인가요? | 가장 형식적. "주된"이 약간 부담. |

→ **권장 B**.

### Helper text (헤드 아래 작은 글씨)

| # | Helper |
|---|--------|
| **A (권장)** | **선택은 진행 소식을 더 잘 보내드리는 데만 써요** |
| B | 선택은 진행 소식 맞춤에 활용해요 |
| C | 선택해주시면 더 맞는 소식을 드릴 수 있어요 |

→ **권장 A**. "더 잘 보내드리는 데만 써요"가 *용도 한정*을 명확히 — PIPA 목적 외 이용 금지 정신을 카피로 표현.

### 3.4 약관·개인정보 처리 안내 링크 텍스트

본 라운드는 *링크 위치만* 결정. 실제 페이지 본문은 별도 라운드.

| 슬롯 | 후보 | 권장 | 사유 |
|------|------|------|------|
| 약관 | "약관" / "이용약관" / "서비스 이용약관" | **이용약관** | "약관" 단독은 모호. "서비스 이용약관"은 길고 형식적. |
| 개인정보 | "개인정보 처리 안내" / "개인정보처리방침" / "프라이버시" | **개인정보 처리방침** | PIPA 제30조 법정 문서명 일치. "프라이버시"는 영어 잔존. |
| 약관 버전 | "약관 버전: 2026-05-04" 등 | **약관 적용일: 2026-05-04** | "버전"은 SW 인상. "적용일"이 법적 문서 톤. |

#### 표기 위치 권장

체크박스 아래 한 줄 작은 글씨:
```
[이용약관] · [개인정보 처리방침] · 약관 적용일: 2026-05-04
```

가운뎃점(`·`) 구분이 시각적으로 가장 자연스럽다. 디자이너 §3에서 회색 톤·48×48 hit area 적용.

#### 3.5.3 거절 시 동적 변화 — 확인 버튼 라벨

체크박스 해제 상태에서 Primary 버튼 라벨이 *바뀔지*:

- "확인하고 신청"이 옵트인·거절 양쪽에 자연스러움 (§3.5.1).

### 3.6 약관 버전 표시

### 5.2 신규 추가 케이스

#### 5.2.1 `duplicate` — 같은 이메일 재제출

→ **권장: 사용자에게는 success로 처리**(13.md §9.6 + 12 §3.2 upsert 정책). 별도 카피 *불필요*. 내부 로깅만(`form_submit_success`에 `is_duplicate: true` property 추가 권장 — 09에서 처리).

#### 5.2.2 `consent_required` — RLS sanity 검증 실패

(`consent_marketing=true`인데 `consent_at=null` 같은 비정상 케이스)

| # | 메시지 | 코멘트 |
|---|------|--------|
| **A (권장)** | **잠깐 문제가 있었어요. 다시 한 번 시도해주세요.** | 사용자에게 *기술 디테일 누설 X*. 친근 어조 유지. |
| B | 동의 정보 처리에 문제가 있었어요. 다시 시도해주세요. | "동의" 단어 노출 = 사용자 혼란 가능. |
| C | 일시적인 오류가 발생했어요. 다시 시도해주세요. | 형식적. |

→ **권장 A**. *원인을 사용자에게 노출하지 않으면서* 재시도 유도.

> **카피 메모 (구현자에게)**: 본 케이스는 RLS sanity가 막아서 발생. 재시도 시에도 같은 에러면 콘솔 경고 + Sentry/PostHog로 운영자 알림(별도 분석 라운드).
