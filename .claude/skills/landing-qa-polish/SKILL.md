# Landing QA Polish

빌드 직후의 사이트는 *대부분 80% 완성*이다. 마지막 20%가 *세상 어디서도 보지 못할 퀄리티*를 만든다. 이 스킬은 그 20%를 *체계적으로* 발견·수정하는 방법론. `landing-qa-polisher` 에이전트가 사용한다.

## 왜 이 스킬이 필요한가

QA의 흔한 함정:
1. 코드만 읽고 "괜찮아 보인다"
2. 데스크탑만 검수
3. 디자인 일관성·마케팅 효과 검증 누락
4. 발견 이슈를 *해결 책임 분담* 없이 보고만
5. "더 임팩트 있게" 같은 모호한 피드백

이 스킬은 *실제 브라우저 검수* + *명확한 분담* + *구체적 수정안*을 강제한다.

## 3축 검증

모든 QA 사이클은 다음 3축을 *모두* 점검:

### 축 1: 기능 QA
- 모든 CTA 클릭 작동
- 폼 검증·에러·성공 상태
- 콘솔 에러·경고 없음
- 4개 디바이스 사이즈에서 깨짐 없음
- 60fps 인터랙션

### 축 2: 디자인 일관성 검수
- 색·폰트·간격이 시안과 *정확히* 일치
- 디테일에 균열 없음 (1px 어긋남, 호버 누락 등)
- 모든 컴포넌트가 디자인 토큰 사용
- 모션 사양 준수 (duration, easing)

### 축 3: 마케팅·전환 가설 검수
- **5초 룰**: 5초만 보고 (a) 무엇을 파는가 (b) 누구를 위한 것인가 (c) 다음 행동이 뭔가 답할 수 있나
- CTA가 fold 안에 보이나
- 가격 정당화 명확한가
- 신뢰 시그널 충분한가
- 페르소나 적합성 (한지원이 보면 "내 얘기야"라고 느끼나)

## 디바이스 매트릭스 — 최소 4개

| 디바이스 | 사이즈 | 주의점 |
|--------|------|------|
| iPhone SE | 375×667 | 작은 화면, 한국 사용자 많음 |
| iPhone 14 Pro | 393×852 | 모던 iOS Safari + 노치 |
| 1280×800 | 노트북 | 중간 데스크탑 |
| 1920×1080 | 풀 데스크탑 | 큰 화면 임팩트 |

(여유 있으면 Pixel 7, iPad도 추가)

각 사이즈에서:
1. Hero 첫 fold 스크린샷 (5초 룰)
2. 전체 페이지 스크롤 영상 또는 연속 스크린샷
3. 시그니처 인터랙션 트리거 → 캡처
4. CTA 클릭 → 응답 캡처

## 5초 룰 — 명시적 검증

검증 방법:
1. 사이트를 처음 보는 척 5초만 본다 (실제로 5초 타이머)
2. 즉시 답한다:
   - **무엇을 파는가**: ___
   - **누구를 위한 것인가**: ___
   - **다음 행동은**: ___
3. 답이 모호하거나 틀리면 → *Hero 구조 문제*

답이 다음과 일치해야 통과:
- "8주 위 건강 코칭 (또는 식습관 코칭)"
- "혼밥 직장인·다이어터·위염 환자"
- "8주 코스 시작 또는 베타 합류"

다르면 Hero 카피·디자인을 storyteller·designer에게 *재작업 요청*.

## 직접 수정 vs 재작업 요청 결정 매트릭스

| QA가 직접 수정 | 구현자 재작업 요청 | 디자인·카피 팀 재작업 요청 |
|------------|------------|------------------|
| 오타·문법 | 컴포넌트 재구조 | 카피 톤 변경 |
| 작은 간격·정렬 | 라이브러리 교체 | 디자인 시스템 변경 |
| 색 대비 미세 조정 | 인터랙션 재구현 | 정보 흐름 재배치 |
| `aria-label` 누락 | 성능 최적화 (큰 변경) | 5초 룰 미통과 |
| 모바일 텍스트 줄바꿈 | 빌드 설정 변경 | 페르소나 메시지 약함 |

직접 수정 시 *반드시* 재빌드·재검증.

## 브라우저 자동화 검수

사용 가능한 스킬: `browse`, `gstack`, 또는 Playwright 직접 호출.

기본 흐름:
```bash
# 1. 사이트 서빙
cd landing
npm run preview &  # 또는 npm run dev
sleep 3  # 서버 시동 대기

# 2. browse 스킬로 4개 사이즈 캡처
# (browse는 navigate + setViewport + screenshot 명령 제공)
```

자동화 체크 항목:
- 콘솔 로그 캡처 (에러·경고)
- Network 탭 — 4xx·5xx 없음
- Performance — LCP·CLS·INP 측정
- a11y — Axe 자동 검사 (`axe-core`)

## 폴리시 디테일 체크리스트

세계 수준 사이트와 평범한 사이트의 차이는 *디테일*:

- [ ] 호버 transition duration이 모든 요소에 일관
- [ ] 폰트 weight 0이 아닌 적절한 무게 (특히 한글 — 400 vs 500 차이가 큼)
- [ ] 줄간격이 한글에 맞춰 조정 (영문 1.5 → 한글 1.6-1.7 권장)
- [ ] 스크롤바 스타일 (Webkit 스크롤바 숨김 또는 커스텀)
- [ ] 포커스 ring 디자인 (기본 outline 못생김 → 커스텀)
- [ ] 페이지 전환 시 깜빡임 없음
- [ ] 이미지 lazy load 시 placeholder
- [ ] 폰트 로딩 시 점프 없음 (font-display: swap + size-adjust)
- [ ] 다크 모드 대응 (필수 X, 있으면 가산)
- [ ] OG 이미지 미리보기 정상 (Slack·KakaoTalk·Twitter 시뮬)
- [ ] favicon 32×32 + 192×192 + 512×512
- [ ] meta description, og:description 작성

## 흔한 실수

- ❌ Lighthouse 점수만 보고 *경험 품질* 검증 안 함
- ❌ "구현자가 잘 만들었다"는 가정으로 검수 시작 (검수자는 *항상* 의심)
- ❌ 폴리시 후 재검증 없이 다음으로 (수정이 다른 곳을 깰 수 있음)
- ❌ 모호한 피드백 ("좀 더 임팩트 있게") — 구체적 변경안으로
- ❌ 한 번에 5개 수정 후 재검증 (어디서 깨졌는지 모름)
