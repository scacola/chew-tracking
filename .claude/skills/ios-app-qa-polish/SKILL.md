# iOS App QA & Polish

빌드 통과 ≠ 작동 앱. 사용자 경험·신호 정확도·접근성·코칭 톤이 모두 정합해야 한다. QA의 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"** — 신호 사양과 실제 동작, UX 사양과 실제 화면, 데이터 모델과 대시보드 표시값을 *동시에 읽고 shape 매칭*. `ios-app-qa` 에이전트가 사용한다.

## 왜 이 스킬이 필요한가

iOS QA의 흔한 함정:
- 코드만 읽고 "괜찮아 보인다"
- 시뮬레이터 단일 사이즈만 검증
- *옵션 G 톤 가이드 위반* (가짜 권위·의료 약속) 무시
- 발견 이슈를 *해결 책임 분담* 없이 보고만
- 모듈 단위가 아닌 *전체 완성 후 1회* 검수 (회귀 누적 디버깅 불가)
- "더 임팩트 있게" 같은 모호한 피드백
- 시뮬레이터에서 안 되는 것을 *버그*로 분류 (실기기 전용 항목 별도 표시 필요)

이 스킬은 *3축 검수 + 점진적 QA + 명확한 분담 + 구체적 수정안*을 강제한다.

## 3축 검증

모든 QA 사이클은 다음 3축을 *모두* 점검:

### 축 1: 기능·정합성
- 빌드 통과 (`xcodebuild`)
- 단위 테스트 통과 (`xcodebuild test`)
- 04_brief 화면 인벤토리 모두 진입 가능
- Mock 모션 → 데이터 저장 → 대시보드 표시값 일치
- 권한 흐름 사양 순서 일치, 거부 시 fallback 정상
- AirPods 분리 시나리오 사양 일치
- SwiftData 마이그레이션 시뮬레이션 통과

### 축 2: 신호·UX 톤
- 코칭 메시지 라이브러리에 의료 약속·100% 정확·치료 약속 *없음*
- 한국어 톤 가이드 준수 (다노식 친근 + 임상 권위 균형, 영어 기계번역 톤 없음)
- 결과 프레이밍 ("씹기 트래커" 표현 회피, 위 건강·체중 결과 언어)
- 정확도 컨텍스트가 필요한 곳에 표시 ("추정 정확도 ±15%")
- 5초 룰 — 첫 실행 / 첫 식사 결과 모두 통과
- 영상 시청 컨텍스트에 거슬리지 않는 피드백

### 축 3: 품질·접근성
- 다크 모드 + Light 모드 모두 정상
- Dynamic Type Large(XL) 깨짐 없음
- VoiceOver로 핵심 흐름 (온보딩 → 식사 시작 → 대시보드) 가능
- WCAG AA 색 대비 (텍스트 4.5:1 이상)
- prefers-reduced-motion 시 모션 비활성화
- 빌드 워닝 0개 또는 사유 명시
- 단위 테스트 — 신호 처리 알고리즘 핵심 케이스 커버

## 디바이스 매트릭스 — 최소 4개

시뮬레이터로:

| 디바이스 | 사이즈 | 주의점 |
|--------|------|------|
| iPhone SE (3rd gen) | 작은 화면 | 한국 사용자 다수, 한 손 thumb zone 좁음 |
| iPhone 15 | 표준 | 모던 iOS 17, 노치 |
| iPhone 15 Pro Max | 큰 화면 | 한 손 reachability |
| iPad (Air 5th gen) | 태블릿 | 가로/세로 둘 다 |

각 사이즈에서:
1. 첫 실행 → Onboarding (5초 룰)
2. ActiveMeal 화면 (한 손 thumb zone)
3. Dashboard (스크롤 + 차트)
4. 다크 모드 + Dynamic Type XL

스크린샷은 `_workspace/app/screenshots/`에 저장.

## 5초 룰 — 명시적 검증

### 첫 실행 5초 룰
1. 앱 첫 진입 5초만 본다 (실제 5초 타이머)
2. 즉시 답한다:
   - **무엇을 하는 앱인가**: ___
   - **누구를 위한 것인가**: ___
   - **다음 행동은**: ___
3. 답이 다음과 일치해야 통과:
   - "AirPods로 식사 패턴을 자동으로 보는 앱"
   - "위 건강·식습관 신경 쓰는 사람"
   - "권한 동의 → 첫 식사"

### 첫 식사 결과 5초 룰
1. Mock 모션으로 가상 식사 종료 → 결과 화면 5초 본다
2. 답한다:
   - **오늘 결과가 무엇인가**: ___
   - **어떻게 변화하고 있는가**: ___
3. 답이 *결과 언어* (시간·천천히/빨리·점수)로 답할 수 있어야

답이 모호하면 → Hero 카드/Dashboard 카피·정보 위계 *재작업 요청*.

## 옵션 G 톤 가이드 위반 적발

코드 + 카피를 grep으로 적극 적발:

```bash
# 의료 약속 적발
rg -n -i "치료|완치|보장|100%|정확.*100" app/

# 영어 잔존 적발
rg -n -i '"(track|stats|monitor|coach[^a-z])' app/

# 도구 프레이밍 적발 (사용자 노출 카피 한정)
rg -n -i "씹기 횟수 측정|chewing tracker" app/
```

발견 시:
- 카피 단순 수정 → *직접 폴리시*
- 시스템 메시지 구조 자체 문제 → *디자이너 재호출*

## 직접 수정 vs 재작업 요청 결정 매트릭스

| QA 직접 수정 | 구현자 재작업 | 디자이너 재작업 | 신호 엔지니어 재작업 |
|------------|------------|---------------|------------------|
| 오타·문법 | 컴포넌트 재구조 | 카피 톤 변경 | 알고리즘 정확도 |
| 작은 간격·정렬 | 상태 관리 버그 | 정보 위계 변경 | KPI 미달 |
| 색 대비 미세 조정 | 라이브러리 교체 | 화면 흐름 변경 | 식사 외 활동 분리 실패 |
| accessibilityLabel 누락 | 권한 흐름 재구현 | 5초 룰 미통과 | 캘리브레이션 전략 변경 |
| 영어 잔존 카피 1-2건 | SwiftData 마이그 | 코칭 메시지 라이브러리 부족 | false positive 폭증 |

직접 수정 시 *반드시* 재빌드·재검증.

## 점진적 QA 운영

**전체 완성 1회 검수 금지**. 모듈 단위로:

- Step 1 (데이터 모델) 완성 → 즉시 Repository CRUD 검증
- Step 2 (코어 알고리즘) 완성 → 즉시 Mock 시나리오 + KPI 비교
- Step 3 (UI 화면) 각 화면 완성 → 즉시 시뮬레이터 진입 + 5초 룰
- Step 4 (라이브 모션) 완성 → 시뮬레이터 빌드 통과 + 실기기 항목 표시
- Step 5 (코칭 메시지) 완성 → 톤 가이드 grep
- Step 6 (폴리시) 완성 → 디바이스 매트릭스 + Dynamic Type + VoiceOver

매 라운드 결과를 06_qa_report에 *추가* (덮어쓰지 않음).

## 코드 리뷰 체크리스트 (코드 직접 읽기)

빌드만 통과해도 못 잡는 이슈를 *코드 리뷰*로 잡는다:

- [ ] `@MainActor` / `Sendable` 누락이 워닝 0인지
- [ ] SwiftData `@Model` 변경 시 마이그레이션 정의 존재
- [ ] `MockMotionStream` / `LiveMotionStream` 분기 정확
- [ ] 권한 거부 시 빈 화면 또는 크래시 대신 *안내 화면*
- [ ] 한국어 placeholder 잔존 (영어 not localized) 없음
- [ ] 색·폰트가 디자인 토큰만 사용 (인라인 색 없음)
- [ ] async/await 캔슬레이션 처리 (식사 종료 시 stream cancel)
- [ ] 세션 비정상 종료 시 부분 데이터 저장

## 흔한 실수

- ❌ 모듈 단위가 아닌 전체 완성 후 1회 검수
- ❌ 시뮬레이터에서 안 되는 것을 *버그*로 분류 (실기기 전용 항목 미표시)
- ❌ 카피 오타를 재작업 요청으로 (직접 폴리시 가능)
- ❌ 알고리즘 정확도 이슈를 직접 수정 (신호 엔지니어 재호출)
- ❌ 옵션 G 톤 위반 무시
- ❌ 5초 룰을 사용자 시점이 아닌 *개발자 시점*으로 평가
- ❌ 회귀 위험 영역 미명시 → 다음 라운드 새 이슈 폭증
- ❌ "더 임팩트 있게" 같은 모호한 피드백
