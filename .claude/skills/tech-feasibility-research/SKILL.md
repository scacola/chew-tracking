# Tech Feasibility Research

웨어러블 IMU·센서 기반 활동 인식 기술의 타당성을 객관적 근거로 검증하는 방법론. `tech-feasibility-researcher` 에이전트가 사용한다.

## 왜 이 스킬이 필요한가

기술 타당성 조사는 "가능한가/불가능한가"의 이분법으로 끝나기 쉽다. 이 스킬은 그것을 막는다. 모든 결론은 *조건부*이며, 그 조건을 명시적으로 분리해야 의사결정에 쓸모가 있다.

## 조사 4축

모든 기술 타당성 조사는 다음 4축을 모두 다뤄야 한다:

1. **하드웨어 접근성** — 센서에 *제3자가* 접근할 수 있는가? 권한 모델, OS 제한, 디바이스 보급률.
2. **신호 품질** — 검출 대상(여기서는 저작운동)이 센서 신호에 *충분히 나타나는가*? 노이즈 비율, 샘플링 레이트 충분성.
3. **알고리즘 성숙도** — 학계/산업계가 *이미 푼 문제*인가, 아니면 우리가 처음 푸는가? 정확도 baseline 존재 여부.
4. **운영 비용** — 매일 사용 가능한 수준의 배터리/CPU/저장 부담인가? 데이터 프라이버시 처리 가능한가?

조사 시 4축 중 하나라도 빠지면 결론이 부풀어진다.

## 출처 신뢰도 라벨

조사 시 모든 출처에 라벨을 붙인다 (보고서에는 출처별로 명시):

| 라벨 | 의미 | 예시 |
|------|------|------|
| 🟢 **1차 (peer-reviewed)** | IEEE/ACM/PubMed 게재 | "Smith et al. 2023, IEEE Sensors" |
| 🟡 **2차 (공식)** | 제조사 공식 문서 | Apple Developer Docs |
| 🟠 **3차 (산업)** | 산업 보고서, 화이트페이퍼 | Gartner, IDC |
| 🔴 **4차 (비공식)** | 블로그, 트윗, 추측 | 개인 블로그, 미디어 보도 |

🔴 출처만으로 결론을 내리지 않는다. 🔴는 "이런 의견도 있다"의 증거로만 쓴다.

## 검색 전략

### 학술 검색 (영문)
- Google Scholar: `chewing detection IMU`, `mastication monitoring wearable`, `eating episode detection accelerometer`
- IEEE Xplore: `inertial sensor mastication`
- PubMed: `eating speed monitoring sensor`
- arXiv: `human activity recognition chewing`

검색 시 *최근 5년*을 우선하되, 핵심 기반 논문(예: 첫 chewing detection paper)은 시기 무관 포함.

### 공식 문서
- Apple: `CMHeadphoneMotionManager`, `Core Motion AirPods`
- Google: `Wear OS Motion API`
- 제조사: Bose, Sony, 삼성 갤럭시 버즈 SDK

### 오픈소스
- GitHub 검색: `chewing detection`, `eating activity recognition`, `mastication`
- 데이터셋: PAMAP2, USC-HAD에서 식사 관련 라벨 확인

### 한국어
- 학술: KISS, RISS — `저작운동 측정`, `식사 활동 인식`, `웨어러블 식사 모니터링`
- 산업: 한국전자통신연구원(ETRI) 보고서

## 출력 스키마 — 정확도 표

연구 결과를 인용할 때 다음 표 형식을 사용:

```markdown
| 연구 | 센서 | 표본 | 정확도 | 조건 | 출처 라벨 |
|------|------|------|--------|------|----------|
| Smith 2023 | 안경형 가속도계 | 30명 / 실험실 | F1 0.89 | 정해진 식사만, 걷기 제외 | 🟢 |
| Lee 2024 | 이어폰 IMU | 12명 / in-the-wild | F1 0.71 | 다중 활동 혼재 | 🟢 |
```

표본 수, 환경(실험실/현실), 정확도 지표(F1 vs 정확도)를 반드시 명시. 이게 없으면 수치 자체가 무의미.

## 한계 점검 체크리스트

보고서 작성 전 다음을 모두 확인:

- [ ] 식사가 아닌 활동 (말하기, 걷기, 머리 끄덕임)과 어떻게 구분?
- [ ] 사용자별 보정(calibration) 필요성? 콜드 스타트 비용?
- [ ] 좌우 AirPods 동시 착용 의무? 한쪽만이면?
- [ ] 배터리 영향 (지속 모션 감지 시)?
- [ ] 데이터 프라이버시 (모션 데이터를 어디서 처리?)
- [ ] 안드로이드 / 갤럭시 버즈 호환?

각 항목을 보고서 "결정적 한계" 섹션에 답한다.

## 인접 기술 비교 (필수)

AirPods IMU 단일 접근만 보면 안 된다. 다음 대안을 *반드시* 비교 검토:

- 이어폰 마이크 기반 저작 *소리* 검출
- EMG 패치 (저작근 근전도)
- 광학 센서 (안경에 부착, 턱 움직임 추적)
- 안경형 가속도계
- 손목 시계 IMU (먹는 동작 검출, 일부만 가능)
- 카메라 + CV (고정 카메라 또는 스마트폰)

각 대안에 대해 (정확도, 사용자 마찰, 비용, 폼팩터, 사회적 수용성)을 짧게 평가.

## 흔한 실수

- ❌ "Apple이 못 막을 거야"라고 가정하고 비공식 API 활용 시나리오 고려 — 앱스토어 통과 못 함
- ❌ 실험실 정확도 = 현실 정확도라고 가정
- ❌ 단일 논문의 결과만 보고 결론
- ❌ 학생 표본 N=10 결과를 일반화

## 결론 형식

조사를 마치면 결론은 반드시 다음 셋 중 하나로:

- **GO (현재 기술로 시장 출시 가능)** — 조건과 함께
- **조건부 GO (특정 가정·확장 필요)** — 무엇이 추가로 필요한지
- **NO-GO / PIVOT (기술 한계 결정적)** — 대안 기술 제안 포함

흐릿한 결론은 의사결정에 무용하다.
