# Discovery Synthesis

리서치 4개를 *임원이 결정할 수 있는 보고서*로 합치는 방법론. `discovery-synthesizer` 에이전트가 사용한다.

## 왜 이 스킬이 필요한가

리서치 4개 보고서를 그냥 읽기 어렵다. 의사결정자(여기선 사용자 본인)는 *최종 권고와 그 근거*를 한 문서에서 봐야 한다. 단순 복사-붙여넣기는 무용 — 발견들 사이의 *모순*과 *시너지*를 드러내는 게 합성의 가치다.

## 입력 (모두 필수)

다음 4개 파일 모두 존재해야 시작:
- `_workspace/01_tech_feasibility.md`
- `_workspace/02_competitive_landscape.md`
- `_workspace/03_market_demand.md`
- `_workspace/04_product_ideation.md`

하나라도 없으면 즉시 멈추고 오케스트레이터에 에러 보고. 절대 빠진 영역을 자체 추측으로 메우지 않는다.

## 보고서 구조 (고정)

```markdown
# [도메인] 디스커버리 보고서
*작성일: YYYY-MM-DD*

## Executive Summary
- 한 줄 결론
- 핵심 권고 (GO / 조건부 GO / PIVOT / NO-GO)
- 결정의 3대 근거
- 즉시 다음 행동 1순위

## 1. 결정 매트릭스
[4축 × 신호 강도 × 출처]

## 2. 기술 타당성 요약
[기술 보고서 핵심 발췌, 출처 표시]

## 4. 경쟁 환경 요약

## 5. 추천 제품 컨셉
[제품 보고서 1순위 + 2순위 짧게]

## 6. 핵심 모순·결정 포인트
[발견 사이의 충돌과 의사결정 필요 지점]

[조건별 신호 표]

## 8. 리스크 (우선순위 순)
```

분량 가이드: A4 8-12장. 너무 짧으면 근거 부족, 너무 길면 결정 방해.

## 출처 병기 — 의무

모든 주장은 4개 보고서 중 어느 것에서 왔는지 표시. 표기법 표준화:

```
[기술-3.2]      → tech_feasibility.md 섹션 3.2
[시장-페르소나#2]  → market_demand.md 페르소나 #2
[경쟁-WhiteSpace] → competitive_landscape.md "White Space" 섹션
[제품-옵션B]      → product_ideation.md 옵션 B
```

출처 없는 주장은 본인의 *합성 의견*임을 명시 ("[합성]" 라벨).

## 모순 식별 — 핵심 가치

리서치 보고서 4개 사이에는 보통 몇 가지 *모순* 또는 *긴장*이 있다. 이걸 드러내는 게 합성의 핵심 가치. 흔한 모순 패턴:

| 패턴 | 예시 |
|------|------|
| 기술 ↔ 시장 | 기술 가능성 강하나 시장 페르소나 약함 |
| 시장 ↔ 경쟁 | 시장 큰데 경쟁사가 모두 적자/단종 → 무언가 어려움 |
| 페르소나 ↔ BM | 강한 페르소나의 지불 의향이 낮음 |
| 차별화 ↔ 기술 한계 | 차별화 포인트가 기술 한계로 약화 |

각 모순을 발견하면 "**섹션 6: 모순·결정 포인트**"에 다음 형식으로:

```markdown
### 모순 1: 기술은 가능, 정확도 baseline 부재
- **증거 A** [기술-결론]: AirPods IMU로 저작 검출 가능, CMHeadphoneMotionManager API 활용 가능
- **증거 B** [기술-한계]: 그러나 in-the-wild 정확도 데이터 부재, baseline 미존재
- **결정 필요**: MVP에서 정확도를 어디까지 약속할 것인가? 부정확한 사용자 경험이 신뢰를 무너뜨릴 위험.
- **결정 옵션**: (a) 정확도 약속 안 함, "트렌드만 보여주기" (b) 의료급 정확도 목표 + 인허가 트랙 (c) 사용자별 보정 단계 추가
```

모순을 드러내고 옵션을 제시하되, *최종 결정은 사용자에게* 맡긴다.

## GO/NO-GO 의사결정 프레임

다음 표 형식으로:

```markdown
| 조건 | GO 신호 | NO-GO 신호 | PIVOT 신호 |
|------|---------|-----------|-----------|
| 기술 정확도 | 프로토타입에서 F1 > 0.7 (in-the-wild) | F1 < 0.5 | F1 0.5-0.7 → 폼팩터 변경 |
| 페르소나 페인 | 인터뷰 5명 중 4명+ 강한 페인 | 2명 이하 | 페르소나 다른 그룹으로 전환 |
| 지불 의향 | 5명 중 3명+ 월 3,000원+ 지불 의사 | 0-1명 | 무료 + 광고 BM |
| 경쟁 우위 | 차별화 가설 디스커버리 인터뷰에서 검증 | 사용자가 차별화를 인식 못 함 | 다른 차별화 축 |
```

각 행은 "**다음 단계에서 어떤 데이터가 들어오면 어떤 결정을 할지**"를 *미리* 못 박는 것이다. 이걸 만들어두면 후속 검증 시 결정이 빨라진다.

## 리스크 — 우선순위 정렬

모든 리스크가 동등하지 않다. 정렬 기준:
1. **결정 영향도** — 이 리스크가 *지금 시작할지 말지*에 영향?
2. **발생 가능성**
3. **완화 비용**

```markdown
**리스크 1 (결정 임팩트 ★★★)**: AirPods 한 면 한 면 마이크/모션 데이터 비대칭 — 좌우 동시 데이터 못 받을 수 있음
- 영향: 정확도가 출시 후 평가에 못 미칠 가능성
- 발생 가능성: 중-고
- 완화: MVP 전 프로토타입으로 1주 내 검증 가능 → 검증 우선순위 1순위
- 출처: [기술-한계]
```

상위 3-5개만 자세히. 나머지는 한 줄로.

## 검증 가설 — 다음 단계 입력

제품 보고서의 "검증해야 할 가설"을 *우선순위*로 다시 정렬해서 5-7개로 압축:

```markdown
1. (기술) AirPods IMU로 저작 검출 F1 ≥ 0.7 in-the-wild → 1주 프로토타입
2. (시장) 페르소나 [한지원]형 사용자가 *이미* 천천히 먹기를 시도해본 적 있음 → 5명 디스커버리 인터뷰
3. (지불) 페르소나가 월 3,000-5,000원 지불 의향 → 인터뷰에서 직접 질문
4. (행동) 실시간 피드백(햅틱)이 식사 속도를 실제로 늦춤 → A/B 테스트 (소규모, 본인+지인)
5. (경쟁) 경쟁사 단종 사례에서 사용자가 옮긴 대안 = 디지털 솔루션 아님 → "그냥 의식하기" → 우리가 [의식 → 자동화]를 제공할 수 있나?
```

각 가설에 *어떻게 검증하는지*, *얼마나 걸리는지* 명시.

## 다음 단계 — 실행 가능 행동

1주, 1개월, 3개월 시간대로 분류:

```markdown
**1주 내**:
- AirPods IMU 프로토타입 — 본인 식사 데이터 수집, 저작 패턴 시각화
- 페르소나 인터뷰 대상자 5명 모집 (지인망 활용)

**1개월 내**:
- 디스커버리 인터뷰 5건 완료, 보고서 작성
- F1 정확도 0.7+ 달성 알고리즘 1차 시도

**3개월 내**:
- MVP TestFlight 배포 (Closed Beta 10명)
- 핵심 가설 1, 2, 3 검증 결과로 Go/No-Go 재평가
```

추상적 "추가 조사" 금지. *돈, 시간, 사람*을 어디 쓸지 적힌 계획.

## 톤 — 컨설팅, 영업 아님

- ✅ "데이터는 X를 시사한다. 그러나 Y라는 결정적 한계가 있다. 우리는 Z를 추천한다."
- ❌ "이것은 훌륭한 기회이며 시장은 폭발적으로 성장 중!"

의견은 명확히, 근거는 객관적으로, 단점은 숨기지 말 것.

## 후속 작업 시 행동

- 사용자가 보고서 피드백 → 해당 섹션만 수정, 나머지 보존
- 입력 보고서 갱신 → 영향받은 섹션 + Executive Summary 갱신
- 보고서 하단에 "**개정 이력**" 섹션 유지 (날짜, 변경 내용, 사유)

## 흔한 실수

- ❌ 입력 보고서를 그대로 복붙
- ❌ 모순 회피 (모든 게 잘 맞는 것처럼 결론)
- ❌ "조심스럽게 추진" 같은 무결정 권고
- ❌ 영업 톤
- ❌ 본인 추측을 출처 표시 없이 섞기
